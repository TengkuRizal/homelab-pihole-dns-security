# Threat Model

## Overview

This document describes the basic threat model for the homelab Pi-hole DNS security service.

The purpose of this threat model is to identify key assets, possible risks, potential impact, and mitigation controls related to running Pi-hole as a DNS filtering layer.

## Assets

| Asset | Description |
|---|---|
| Pi-hole DNS service | Main DNS filtering service for selected clients |
| Raspberry Pi host | Physical device running Pi-hole |
| Pi-hole admin interface | Web console used to manage Pi-hole |
| DNS query data | DNS activity generated by clients |
| Blocklist and allowlist configuration | Filtering policy used by Pi-hole |
| Backup configuration | Recovery data for restoring Pi-hole |
| Network clients | Devices depending on DNS resolution |

## Threats

| Threat | Impact | Mitigation |
|---|---|---|
| Pi-hole service unavailable | Clients may fail to resolve domains | Runbook, service restart steps, fallback DNS plan |
| Raspberry Pi hardware failure | DNS filtering service becomes unavailable | Backup export, restore documentation, future secondary Pi-hole |
| Admin interface exposed | Unauthorized configuration changes | Restrict access to trusted local network only |
| Weak admin password | Unauthorized access to Pi-hole dashboard | Use strong password and avoid sharing credentials |
| False positive domain block | Legitimate application or website breaks | Allowlist review process and validation |
| Aggressive blocklist | Excessive blocking and user disruption | Use reputable blocklists and validate changes |
| Raw DNS logs exposed | Privacy leakage and browsing pattern exposure | Do not publish raw logs, sanitize screenshots |
| Client bypasses Pi-hole | Reduced DNS visibility and filtering | Future firewall rule to enforce DNS through Pi-hole |
| Upstream DNS unavailable | Allowed queries may fail | Use reliable upstream resolver and document fallback |
| Misconfiguration | DNS outage or incorrect filtering | Document configuration and backup before changes |

## Risk Summary

| Risk | Likelihood | Impact | Priority |
|---|---|---|---|
| Pi-hole service down | Medium | High | High |
| Raspberry Pi failure | Medium | High | High |
| False positive blocking | Medium | Medium | Medium |
| DNS log exposure | Low | High | High |
| Admin interface misuse | Low | High | High |
| Client DNS bypass | Medium | Medium | Medium |

## Key Controls

The following controls reduce operational and security risk:

- Pi-hole dashboard should only be accessible from trusted internal network
- Strong password should be used for admin access
- Raw DNS logs must not be published
- Screenshots must be sanitized before adding to GitHub
- Pi-hole configuration should be backed up using Teleporter
- DNS failure recovery steps should be documented
- Blocklist and allowlist changes should be reviewed
- Future improvement should include secondary Pi-hole or DNS failover

## Assumptions

This threat model assumes:

- Pi-hole is deployed inside a trusted homelab network
- The Raspberry Pi is not directly exposed to the public internet
- Only selected internal clients use Pi-hole as DNS resolver
- The repository is public, so sensitive data must be excluded
- This deployment is for learning, portfolio, and homelab security improvement

## Out of Scope

This threat model does not cover:

- Full enterprise DNS security architecture
- Malware analysis
- Deep packet inspection
- Endpoint detection and response
- Public authoritative DNS hosting
- SIEM correlation
- Advanced threat hunting

## Future Improvements

Potential future improvements:

- Deploy secondary Pi-hole for high availability
- Add firewall rule to prevent DNS bypass
- Monitor DNS service availability
- Export metrics to Prometheus and Grafana
- Automate configuration backup
- Review blocklist effectiveness periodically
- Document incident examples from DNS failures or false positives
